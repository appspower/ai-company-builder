## 1. 프로젝트 한 줄 정의

**사용자가 직접 AI 직원으로 구성된 가상 회사를 만들고, 업무를 지시하고, 결재하며 운영하는 SaaS 플랫폼.**

## 2. 가상 보도자료 (PR/FAQ)

### PRESS RELEASE

**"AI Company Builder, 누구나 5분 만에 AI 직원 회사를 세팅하다"**

서울 — AI Company Builder는 오늘, 비개발자도 AI 에이전트로 구성된 가상 조직을 직접 설계하고 운영할 수 있는 SaaS 플랫폼을 출시했습니다.

그동안 AI 에이전트 시스템은 개발자가 코드로 구성하거나, 범용 워크플로우 빌더에서 노드를 연결하는 방식이었습니다. 이 방식은 "회사를 운영하는 CEO"에게 직관적이지 않았습니다.

AI Company Builder는 다릅니다. 사용자는 실제 회사를 세우듯 **부서를 만들고, 직원(AI 에이전트)을 배치하고, 업무를 던지고, 결재하는** 경험을 합니다. 각 AI 직원에게는 고유한 역할, 성격(페르소나), 참고 문서(RAG), 그리고 적합한 AI 모델이 부여됩니다. CEO는 업무 진행 상황을 칸반 보드에서 실시간으로 확인하고, 중요한 단계에서 직접 승인 또는 반려합니다.

"코드 한 줄 모르는 사장님도 AI 팀을 꾸려서 돌릴 수 있어야 합니다. 그게 이 플랫폼의 존재 이유입니다."

### FAQ

**Q. 기존 AI 에이전트 플랫폼(Dify, CrewAI, n8n 등)과 뭐가 다릅니까?**
A. 기존 플랫폼은 "워크플로우 빌더" 또는 "개발자용 프레임워크"입니다. 우리는 **"회사 경영"이라는 메타포**를 씌웁니다. 부서, 직원, 업무 지시, 결재 — CEO가 이미 알고 있는 개념으로 AI 시스템을 조작합니다. 노드를 연결하거나 코드를 쓸 필요가 없습니다.

**Q. 하나의 계정에서 여러 회사를 운영할 수 있습니까?**
A. 네. Multi-Workspace 구조로, 사업별로 독립된 회사를 생성합니다. 각 회사의 데이터, 에이전트, 산출물은 완전히 격리됩니다.

**Q. AI가 실수하면 어떻게 합니까?**
A. Human-In-The-Loop(HITL) 결재 시스템이 있습니다. 중요 단계에서 CEO의 승인 게이트를 설정할 수 있습니다. AI가 초안을 만들면 CEO가 검토하고 승인해야 다음 단계로 넘어갑니다.

**Q. 어떤 AI 모델을 사용합니까?**
A. 업무 난이도에 따라 여러 모델을 혼용합니다. 복잡한 기획은 Claude Opus, 빠른 텍스트 처리는 Sonnet/Haiku, 이미지 생성은 Gemini Flash Image 등 — 에이전트별로 최적의 모델을 지정합니다.

## 3. 타겟 사용자

### Primary: 1인 또는 소규모 사업 운영자 (CEO)

- 여러 사업을 동시에 운영하며, 반복적인 실무를 AI에게 위임하고 싶은 사람
- 코딩을 모르지만, 조직 구성과 업무 지시에는 능숙한 사람
- AI 에이전트의 결과물을 직접 검토하고 최종 의사결정을 내리고 싶은 사람

### Secondary: AI 자동화에 관심 있는 소규모 팀

- 콘텐츠 제작, 이커머스, 마케팅 등 반복 업무가 많은 팀
- 팀 내에서 AI 에이전트를 "가상 팀원"으로 활용하려는 팀

### 명시적으로 타겟이 아닌 사용자

- 대규모 엔터프라이즈 (복잡한 권한 체계, SSO, 감사 로그 등은 V1 범위 밖)
- AI/ML 개발자 (이 플랫폼은 코드 없이 조작하는 것이 핵심)

## 4. 핵심 원칙 (Non-Negotiables)

이 원칙은 모든 기능, 디자인, 기술 결정의 판단 기준입니다. 충돌 시 이 원칙이 우선합니다.

### 원칙 1: "회사 경영" 메타포의 일관성

모든 UI와 개념은 실제 회사 운영의 언어를 사용합니다.
- 에이전트 = 직원, 워크스페이스 = 회사, 미션 = 업무 지시, 승인 = 결재
- 기술 용어(agent, workflow, pipeline)를 사용자에게 노출하지 않습니다.
- CEO가 "이건 내 회사를 관리하는 화면이다"라고 느껴야 합니다.

### 원칙 2: CEO의 통제권 (Human-In-The-Loop)

AI가 자율적으로 실행하되, CEO는 언제든 개입할 수 있어야 합니다.
- 모든 미션에 승인 게이트를 선택적으로 삽입할 수 있습니다.
- CEO가 승인하지 않으면 다음 단계로 넘어가지 않습니다.
- 자동 실행과 수동 결재의 비율은 CEO가 결정합니다.

### 원칙 3: 동적 구성 (Dynamic, Not Hardcoded)

부서, 에이전트, 워크플로우 — 모든 것을 사용자가 직접 만들고 수정합니다.
- 하드코딩된 부서 템플릿에 갇히지 않습니다.
- "이 회사에는 소싱팀이 필요하고, 저 회사에는 기획팀이 필요하다"를 사용자가 자유롭게 정의합니다.
- 에이전트의 역할, 성격, 지식, AI 모델까지 개별 세팅 가능합니다.

### 원칙 4: 산출물 중심 (Artifact-Centric)

에이전트의 가치는 대화가 아니라 산출물로 증명됩니다.
- 모든 미션의 결과는 추적 가능한 파일(산출물)로 남습니다.
- 산출물은 버전 관리되고, 미션/에이전트로 역추적 가능합니다.
- CEO는 "이번 주에 우리 AI 회사가 뭘 만들었나"를 한눈에 봅니다.

### 원칙 5: 단순함 우선 (Simplicity First)

기능의 풍부함보다 사용의 직관성이 우선입니다.
- 화면은 3단계를 넘기지 않습니다 (로비 → 회사 → 상세).
- 고급 기능은 존재하되, 기본 경로는 단순해야 합니다.
- "5분 안에 첫 번째 AI 회사를 세팅하고, 첫 미션을 던질 수 있다"가 목표입니다.

## 5. 명시적으로 하지 않을 것 (Out of Scope — V1)

| 하지 않을 것 | 이유 |
|---|---|
| 에이전트 간 자율 토론/토의 시스템 | V1은 "지시 → 실행 → 결재" 흐름에 집중. 토론은 복잡도가 급증하므로 V2 이후 |
| 실시간 화상/음성 인터페이스 | 텍스트 기반 명령과 산출물 중심. 음성은 별도 프로젝트 |
| 엔터프라이즈 기능 (SSO, RBAC, 감사 로그) | 타겟 사용자가 1인/소규모. 복잡한 권한 체계는 V2 이후 |
| 자체 LLM 호스팅 | 외부 API (Claude, GPT, Gemini) 활용. 모델 호스팅은 범위 밖 |
| 모바일 앱 | 데스크톱 웹 우선. 반응형은 지원하되 전용 앱은 V2 이후 |
| 외부 SaaS 연동 (Slack, Notion, Google Drive) | V1은 플랫폼 내부 완결. 외부 연동은 V2에서 스킬로 확장 |
| 결제/과금 시스템 | V1은 자체 사용 목적. 상용화 과금은 별도 단계 |

## 6. 성공의 정의

### V1 (Phase 1 — Frontend)의 성공

> 더미 데이터 기반으로 4개 워크스페이스의 전체 사용자 흐름을 브라우저에서 체험할 수 있다.
> CEO가 화면을 보며 "이게 내가 원하던 것"이라고 느낀다.

구체적으로:
- 워크스페이스 로비에서 회사를 선택해 입장할 수 있다
- 홈 대시보드에서 승인 대기 건과 전체 현황을 확인할 수 있다
- 조직 & 에이전트 화면에서 부서/에이전트 구성을 시각적으로 확인할 수 있다
- 미션 센터에서 칸반보드로 업무 진행 상태를 확인하고, 승인/반려를 체험할 수 있다
- 산출물 보관함에서 결과물을 필터링하고 미리보기할 수 있다

### V1 Complete (Phase 1 + 2)의 성공

> 실제 AI 모델이 연결되어, CEO가 미션을 던지면 에이전트가 실행하고 산출물을 만들어낸다.
> CEO는 중간 단계에서 승인/반려하며 결과물의 품질을 통제한다.

### 궁극적 비전

> 1인 사업자가 AI Company Builder 하나로 여러 사업의 반복 업무를 AI 팀에게 위임하고,
> 본인은 전략적 의사결정과 최종 품질 관리에만 집중하는 세상.

## 7. 경쟁 환경에서의 포지셔닝

```
                    비개발자 친화적
                         |
            AI Company   |
             Builder ----+---- Relevance AI
              (우리)     |
                         |
   조직 메타포 ----------+---------- 워크플로우 메타포
                         |
                         |
              CrewAI ----+---- Dify.ai
             Enterprise  |      n8n
                         |
                    개발자 중심
```

**우리의 포지션:** 비개발자 친화적 + 조직(회사) 메타포
**핵심 차별점:** "워크플로우를 짠다"가 아니라 "회사를 운영한다"는 경험

## 부록: 용어 정의

플랫폼 전체에서 일관되게 사용할 용어입니다.

| 내부 기술 용어 | 플랫폼 표시 용어 | 설명 |
|---|---|---|
| Workspace | **회사** | 독립된 사업 단위 |
| Department | **부서** | 회사 내 팀 단위 |
| Agent | **직원** (또는 AI 직원) | 특정 역할을 수행하는 AI 에이전트 |
| Workflow | **업무 흐름** | 미션의 단계 순서 |
| WorkflowRun / Mission | **미션** | CEO가 내린 구체적 업무 지시 |
| HITL Approval Gate | **결재** | CEO 승인/반려 단계 |
| Artifact | **산출물** | 에이전트가 만든 파일/결과물 |
| RAG Knowledge Base | **참고 자료** (또는 업무 매뉴얼) | 에이전트에게 주입하는 문서 |
| LLM Model | **AI 엔진** | 에이전트가 사용하는 AI 모델 |
| Persona | **성격** (또는 캐릭터) | 에이전트의 업무 스타일/톤 설정 |
